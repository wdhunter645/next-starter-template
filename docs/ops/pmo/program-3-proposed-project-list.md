# Program 3 Proposed Project List

## Purpose

Provide a discussion draft for the next available Programs 1-4 portfolio lane: Program 3.

Program 3 is not an intake bucket. Program 3 is an available rotating portfolio planning/execution lane. It should only receive projects that have been admitted to the execution portfolio, prioritized, and selected by Bill/Atlas.

## Current status

Program 3 is available. It does not currently own active LGFC implementation work.

This document is a planning proposal only. It does not launch Program 3, create child issues, authorize Cursor execution, or mutate repository issue state.

## Selection criteria

A candidate should be proposed for Program 3 only when it is:

- more mature than a raw idea;
- documented enough to discuss as a project;
- likely to be one coherent program group;
- not already owned by the active Program 2 website implementation cycle;
- not already owned by staged Program 1 PMO automation work;
- ready for prioritization discussion.

## Proposed Program 3 theme

Recommended Program 3 theme:

**Community, Fundraising, and Content Growth Operations**

Rationale:

- Program 2 is stabilizing website/content infrastructure.
- Program 1 is stabilizing PMO/agent governance.
- Program 3 can then focus on outward-facing growth and operations using the stabilized website and PMO foundation.

## Program 3 exclusions

Program 3 should not include:

- raw, unreviewed ideas still belonging in Program 5;
- PMO automation governance already staged under Program 1;
- active Program 2 website implementation tasks;
- production secrets or vendor account setup without separate approval;
- broad website redesign outside approved design authority.

## Required readiness work before Program 3 launch

Before Program 3 can launch:

1. Bill/Atlas must select the actual Program 3 theme.
2. Candidate projects must be confirmed as portfolio projects, not Program 5 drafts.
3. Each selected project needs a short design/readiness document.
4. A Program 3 source issue must be created or authorized.
5. A Program 3 implementation plan must define project order, child issue rules, validation, rollback, and closeout.
6. Cursor execution must wait for explicit launch authorization.
