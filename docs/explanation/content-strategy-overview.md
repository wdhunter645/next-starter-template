# LGFC Content Strategy Overview

## Purpose
This document explains the content strategy program for the Lou Gehrig Fan Club and defines why the system is being designed as a story-first publishing model instead of a loose media archive.

## Program Objective
The LGFC content strategy exists to support a high-quality, historically grounded, story-driven website. The objective is not to accumulate media in bulk. The objective is to build a controlled editorial system that:
- organizes Lou Gehrig content into durable story records
- connects approved media to those story records
- supports search, homepage rotation, and future editorial reuse
- supports member contributions without allowing uncontrolled publication drift

## Design Position
The content system is story-first. A story record is the editorial anchor. Media exists to support the story.

The platform is therefore designed around four layers:
1. story inventory
2. media library and media relationships
3. submission intake and review
4. homepage and editorial placement logic

## Why Structure Is Required
An unstructured archive causes predictable failures:
- duplicate stories with no canonical source
- repeated upload of near-identical images
- weak search quality
- homepage repetition around the same anniversary cluster
- inability to distinguish approved historical narrative from incoming submissions
- loss of attribution integrity

## Design Decisions Resolved Here
This documentation set resolves the substantive conflicts raised during PR #820 review.

### Evergreen stories
Evergreen stories are explicitly supported. They are represented by story records where `event_date` may be null. They remain eligible for editorial use outside anniversary windows.

### Media storage model
The story record does not embed full duplicated media metadata as canonical storage. Instead, story records reference approved media records by identifier. Canonical media metadata is stored once in the media library/reference model. This avoids drift between story records and media records.

### Multi-section placement
A story may appear in multiple homepage sections. Base story metadata remains global, but section-specific presentation must be handled through a section-placement relationship model so a story can carry different prominence, order, or display role per section when needed.

### search_text synchronization
`search_text` is a derived field and must be generated by the database layer or deterministic write-path logic. It must not rely on ad hoc manual updates.

### Canonical uniqueness
The rule that only one canonical story may exist per tag must be enforced at the database level, not only described in prose.
